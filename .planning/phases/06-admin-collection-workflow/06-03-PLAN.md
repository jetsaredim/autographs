<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin multipart upload -> controller | Untrusted file bytes and multipart fields enter image validation and storage logic. |
| controller -> OCI/local media | Controller mutates private original objects outside Oracle transactions. |
| controller -> Oracle image metadata | Controller mutates metadata and cleanup events in the source of truth. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-03-01 | Tampering | image primary/delete/replace routes | high | mitigate | Require authenticated same-origin mutations, validate item/image IDs, and update only images owned by the requested item. |
| T-06-03-02 | Information Disclosure | cleanup warnings | high | mitigate | Return redacted admin messages only; tests deny object keys, bucket names, namespaces, Object Storage URLs, image UUID leakage beyond safe response IDs, and original filenames. |
| T-06-03-03 | Repudiation | cleanup and image history | medium | mitigate | Persist `ImageRemoved`, `ImageReplaced`, `PrimaryImageChanged`, and `CleanupChanged` events so the owner can see what changed. |
| T-06-03-04 | Denial of Service | image uploads/replacements | medium | mitigate | Preserve existing body limit and image-content validation; do not add bulk import or unbounded queues. |
| T-06-03-05 | Tampering | cross-system cleanup | high | mitigate | Use explicit compensation flows and durable cleanup events because Oracle and Object Storage do not share a transaction. |
| T-06-03-SC | Tampering | package supply chain | low | accept | No package-manager installs are planned; implementation must use existing media and image crates. |
</threat_model>
