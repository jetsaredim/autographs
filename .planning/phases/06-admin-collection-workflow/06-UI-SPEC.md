# Phase 6 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none |
| Icon library | none; do not add an icon package for Phase 6 |
| Font | `"IBM Plex Sans", "Segoe UI", system-ui, sans-serif` |

Source: `.planning/phases/06-admin-collection-workflow/06-CONTEXT.md`, `.planning/phases/06-admin-collection-workflow/06-RESEARCH.md`, `AGENTS.md`, `controller/static-public/assets/site.css`, and `controller/static-admin/*`.

Design-system contract:
- Use plain static HTML, CSS, and JavaScript under `controller/static-admin/`; do not introduce React, Next.js, Vite, Tailwind, shadcn, browser storage, or a frontend build pipeline in Phase 6.
- Keep the admin UI quiet, utilitarian, and work-focused. It should feel like a single-owner operational console, not a public landing page or marketing surface.
- Use the public site's established sans-serif type, 8px radius, 44px control height, and neutral surface language as the baseline. Replace the Phase 5 seed shell's Georgia/brown-heavy treatment with this static-public visual system.
- Use text labels for buttons and controls. Do not use icon-only controls because no icon library is present and operational clarity is more important than visual compression for this phase.
- Use dedicated redacted DTOs for every status/diagnostic view. The UI must never render Object Storage URLs, object keys, namespaces, bucket names, Oracle internals, image UUIDs, credentials, original filenames, or unpublished public artifacts.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline form help gaps, status-dot gaps, compact metadata pairs |
| sm | 8px | Button label padding rhythm, stacked label/input gaps, tag/status gaps |
| md | 16px | Default grid gaps, form section padding, panel internal padding |
| lg | 24px | Page padding, hub band gaps, major form group spacing |
| xl | 32px | Header-to-content separation, editor/detail split gaps |
| 2xl | 48px | Page-level vertical rhythm on desktop |
| 3xl | 64px | Reserved for public/marketing pages only; avoid inside the admin workflow |

Exceptions:
- Minimum interactive target height is 44px for buttons, links styled as buttons, inputs, selects, and file-upload triggers.
- Compact table/list rows may use a 40px minimum height when the row also contains a separate 44px action target.
- Status dots may be 8px or 10px because they are non-interactive.
- Form textareas use a minimum height of 96px for description/inscription fields and 64px for short notes.

## Typography

Use exactly these four type sizes and two weights in Phase 6 admin CSS.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.35 |
| Heading | 20px | 600 | 1.25 |
| Display | 32px | 600 | 1.1 |

Typography contract:
- Body copy, form values, JSON-safe diagnostics, and history values use 16px/400.
- Labels, breadcrumbs, section captions, badges, table headers, and helper text use 14px; use 600 for labels and 400 for secondary helper text.
- Section headings use 20px/600. The first-screen hub title is the only admin display heading and uses 32px/600.
- Do not scale font size with viewport width. Letter spacing is `0` except all-caps eyebrows may use `0.08em` maximum.
- Long field values in history rows wrap naturally; do not truncate before/after values unless the full value is available in an expanded details region.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #f7f8f6 | Admin page background and quiet empty space |
| Secondary (30%) | #ffffff | Panels, forms, tables, cards, dialogs, status bands |
| Accent (10%) | #25636a | Primary workflow actions, active tab/step state, focus outline, selected primary image |
| Destructive | #b42318 | Destructive actions only |

Accent reserved for: `Add new item`, `Save item`, `Publish changes`, active navigation/step indicators, focus-visible outlines, selected primary-image marker, and successful actionable status links. Do not use accent for every button, every link, decorative borders, or passive metrics.

Supporting colors:
- Text primary: #242826
- Text secondary: #626862
- Border: #d7ddd5
- Muted surface: #eef2ed
- Warning: #9a6700 for pending unpublished changes, cleanup warnings, expired session notices, and full rebuild caution.
- Success: #1f7a4d for healthy provider/controller/publish states.
- Code/diagnostic background: #111827 with #f9fafb text only inside preformatted redacted diagnostics blocks.

Color contract:
- Keep the UI visually neutral and operational; avoid a one-note beige/brown palette.
- Destructive red is reserved for image removal, release pruning repair actions, forced cleanup retry failure acknowledgement, and logout/session-expired destructive confirmations if needed.
- Publish state uses semantic color plus text. Color alone must never be the only signal.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Add new item |
| Empty state heading | No saved items yet |
| Empty state body | Start with the backlog: add an autograph item, upload its images, save it privately, then publish when the batch is ready. |
| Error state | Something did not save. Review the highlighted fields, keep this page open, and try again. If the problem repeats, check the redacted diagnostics panel. |
| Destructive confirmation | Remove image: Remove this image from the item and queue cleanup of the private original? This cannot be undone from the admin UI. |

Additional required copy:
- Session expired: `Your admin session expired. Log in again to continue; unsent form changes are still on this page.`
- Lockout/rate limit: `Too many login attempts. Wait and try again.`
- Pending changes: `{count} saved change(s) have not been published yet.`
- Publish action: `Publish changes`
- Full rebuild action: `Full rebuild`
- Full rebuild confirmation: `Run a full rebuild only for repair or structural changes. Continue?`
- Save success: `Saved privately. Publish changes when this batch is ready for the public site.`
- Publish success: `Published. The public static release is current.`
- Cleanup warning: `Cleanup needs attention. Review the affected item before publishing again.`
- History empty: `No history recorded yet. Changes made after the Phase 6 history update will appear here.`

Copywriting rules:
- Use concrete verbs: `Add`, `Save`, `Publish`, `Find`, `Mark primary`, `Remove image`, `Retry cleanup`.
- Do not say "seed" in the Phase 6 UI except in historical docs; the admin surface is now a collection workflow.
- Do not expose implementation identifiers in user-facing copy unless they are intentionally safe release IDs or item IDs needed for operator troubleshooting.

## Layout Contract

First-screen admin hub:
- After login, `/admin` opens on a status hub, not an editor.
- The hub uses a constrained shell `max-width: 1120px`, `padding: 24px 16px` on mobile, and `padding: 32px 24px` on desktop.
- The first viewport contains: title `Admin hub`, session state/logout, health summary, pending unpublished changes, last publish result, and two primary paths.
- Primary path is visually first and largest: `Add new item`. Secondary path is `Find or modify existing items`.
- The hub must include a compact diagnostics panel with safe/redacted provider modes, controller health, last publish result, pending unpublished changes, cleanup warnings, and live-smoke guidance.

Navigation:
- Use simple top-level tabs or segmented controls: `Hub`, `Add item`, `Items`, `Publish`, `Diagnostics`.
- Keep the current page/state visible. A user should always know whether they are editing a draft, a published item with unpublished changes, or a clean published item.
- On mobile, tabs wrap into a two-column grid or horizontal scroll without overlapping text.

Forms:
- Group metadata fields into four sections: `Identity`, `Story`, `Provenance`, and `Publication`.
- Required fields: title, signer, category, publication status. Optional fields must be visibly optional through layout and helper text only where it prevents mistakes.
- Use a two-column desktop form grid for short fields and one-column mobile layout below 760px.
- The save area is sticky at the bottom of long edit forms on desktop and normal flow on mobile. It contains `Save item`, dirty-state text, and a non-primary `Discard unsaved changes` action only when there are unsaved client-side edits.

Item finder:
- Existing-item maintenance is secondary but competent. Provide search/filter by signer, title, category, tag, and publication status.
- Item rows show title, signer, status, image count, pending-change marker, and last updated date.
- Row actions are text buttons: `Edit`, `History`, and `Publish status` when applicable.

## Interaction Contract

Session and lockout:
- Logged-out state shows only login form, redacted public-safe explanation, and no catalog or diagnostics data.
- Logged-in state shows clear session status and a `Log out` button in the hub header.
- `401` responses transition to an expired-session message without clearing unsent form fields.
- Login failure and lockout messages are plain language and do not reveal whether a password hash, cookie, token, or internal service failed.

Create/edit workflow:
- `Add new item` opens a blank form optimized for backlog entry.
- Saving writes only to the private source of truth. It never triggers static publish automatically.
- After save, keep the user on the item and show next actions: `Add another item`, `Upload more images`, `Publish changes`, and `Back to hub`.
- Existing items hydrate into the same editor. Optional field clears must be expressible in the UI and not confused with unchanged fields.

Image workflow:
- Allow multiple image upload in one item workflow.
- Display uploaded images in upload order with the primary image first.
- Each image tile shows a thumbnail, primary marker, safe alt text field or summary, and text actions: `Mark primary`, `Remove image`.
- Do not add drag/reorder, captions, review queues, or DAM-style management in Phase 6.
- Removing/replacing images requires confirmation, records a visible item event, and surfaces cleanup failure without exposing object keys.

History workflow:
- Every item editor includes a `History` section or tab.
- History entries are reverse chronological and show event type, timestamp, concise summary, and field-level before/after values for meaningful metadata changes.
- Image changes and publication changes appear as history events even when they are not field diffs.
- Pre-existing records without old events show the history empty copy from the copywriting contract.

Publish workflow:
- Hub and publish view show pending unpublished changes before the publish action.
- `Publish changes` triggers the normal incremental publish path.
- `Full rebuild` is visually secondary, warning-colored, and requires confirmation.
- Publish progress uses text states: `Idle`, `Publishing`, `Succeeded`, `Failed`, with timestamp and safe release summary.
- Publish failures keep saved private changes intact and provide next action text: retry publish, inspect diagnostics, or run live smoke guidance.

Cleanup and retention:
- Cleanup is observable. The hub shows `0 cleanup warnings` when clean and a warning summary when action is needed.
- Release retention/pruning status appears in diagnostics or publish view, not as a dominant hub CTA unless there is a warning.
- Pruning/cleanup repair actions are cautious, text-labeled, and never run silently from page load.

## Component Inventory

Approved Phase 6 components, implemented with static HTML/CSS/JS:
- App shell with header, session status, and top-level tabs.
- Status summary cards for controller, provider modes, publish state, pending changes, cleanup, and release retention.
- Primary/secondary/action/destructive buttons with 44px minimum height and 8px radius.
- Form field, textarea, select, file upload, segmented status control, validation message, and sticky save bar.
- Search/filter toolbar and item result list/table.
- Image upload grid with thumbnail tiles and primary marker.
- History timeline/list with before/after diff rows.
- Confirmation dialog or inline confirmation region for destructive actions.
- Redacted diagnostics panel using preformatted JSON only when it is compact and safe.

Disallowed Phase 6 components:
- Marketing hero sections, decorative cards around whole page sections, nested cards, gradient/orb backgrounds, icon-only toolbars, browser-storage draft managers, drag-and-drop image sorting, public account/profile UI, AI suggestion panels, bulk import queues, and role/permission management.

## State And Accessibility Contract

Required UI states:
- Logged out, logged in, session expired, login failed, lockout/rate limited.
- Hub loading, hub healthy, hub degraded, hub error.
- Empty catalog, populated item list, no search results.
- Form pristine, dirty, saving, saved, validation failed, save failed.
- Upload pending, upload succeeded, upload failed, cleanup warning.
- No history, history loading, history loaded, history unavailable.
- No pending changes, pending changes present, publishing, publish succeeded, publish failed.

Accessibility rules:
- Every input has a visible label.
- Every status message uses `role="status"` for non-critical updates or `role="alert"` for blocking errors.
- Focus moves to the first invalid field after validation failure and to the success/status summary after publish completion.
- Focus-visible outline uses #25636a at 2px with 3px offset.
- Do not rely on color alone for status. Pair status color with labels such as `Healthy`, `Pending`, `Failed`, or `Needs attention`.
- All generated text must fit at 320px viewport width without overlapping adjacent controls.
