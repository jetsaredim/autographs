# Rust Controller Walkthrough

The Rust controller is the private admin and publishing service for the static
runtime migration. It currently proves the foundation: authentication, local
catalog writes, private upload handling, static artifact contracts, validated
publishing, a minimal browser admin shell, and a live OCI persistence smoke
harness.

It is the implemented controller foundation for Phase 5, but the final 05-07
production live proof and cutover documentation still need to be recorded
before closing the phase. Phase 6 will add the polished daily-use collection
workflow on top of this foundation.

## Startup

The entry point is [`controller/src/main.rs`](../controller/src/main.rs).

```text
main.rs
  -> ControllerConfig::from_env()
  -> bind TCP listener
  -> routes::router(config)
  -> Axum server
```

The library module index is [`controller/src/lib.rs`](../controller/src/lib.rs).

## Configuration

[`controller/src/config.rs`](../controller/src/config.rs) reads runtime settings:

- Bind address, defaulting to `0.0.0.0:8080`
- Expected public origin for CSRF checks
- Secure-cookie setting
- Admin password or Argon2 password hash
- Operator bearer token
- Readiness booleans for Oracle, media, and static-release configuration

The readiness flags are intentionally coarse. Health responses report whether
categories are configured, never actual secrets, bucket names, or connection
details.

## Authentication

[`controller/src/auth.rs`](../controller/src/auth.rs) maintains a minimal
single-admin authentication state in memory:

- Successful login creates a UUID session token.
- Failed login attempts lock the path for 60 seconds after five failures.
- Passwords can be checked directly for local development or through Argon2
  hashes.
- Logout removes the active session.
- CLI and operator calls can use a bearer token instead of browser cookies.

This is intentionally simple for one-admin personal operation. Restarting the
controller invalidates browser sessions.

## HTTP Routes

[`controller/src/routes.rs`](../controller/src/routes.rs) defines the current
API:

| Route | Purpose |
| --- | --- |
| `GET /health` | Public redacted liveness |
| `GET /admin/api/health` | Redacted configuration readiness |
| `POST /admin/api/login` | Create admin session |
| `POST /admin/api/logout` | Invalidate session and expire cookie |
| `POST /admin/api/items` | Create item |
| `PATCH /admin/api/items/{id}` | Update metadata |
| `POST /admin/api/items/{id}/images` | Upload private original |
| `POST /admin/api/items/{id}/publication` | Change draft or published status |
| `POST /admin/api/publish/incremental` | Build and promote an incremental static release |
| `POST /admin/api/publish/full` | Build and promote a full static release |
| `GET /admin/api/publish/status` | Read the last redacted publish result |

Browser mutations require a valid cookie plus a matching `Origin` or `Referer`.
Bearer-token calls bypass CSRF checks because tokens are explicit credentials
rather than ambient cookies.

Responses are converted into redacted DTOs in
[`controller/src/routes.rs`](../controller/src/routes.rs). They omit bucket
names, namespaces, object keys, direct storage URLs, and original filenames.

## Upload Flow

The private image flow lives in
[`controller/src/routes.rs`](../controller/src/routes.rs):

```text
POST multipart image
  -> authenticate and authorize mutation
  -> allow JPEG, PNG, or WebP only
  -> reject files over 20 MiB
  -> generate image UUID
  -> build UUID-only object key
  -> store bytes privately
  -> attach private image metadata to item
  -> return redacted item response
```

Object keys are generated by
[`controller/src/storage_keys.rs`](../controller/src/storage_keys.rs):

```text
originals/{item-uuid}/{image-uuid}
```

Source filenames such as `secret bucket photo.jpg` remain private metadata and
never appear in storage paths.

## Persistence Abstractions

[`controller/src/catalog.rs`](../controller/src/catalog.rs) defines
`CatalogRepository`. It currently supports:

- Create item
- Update item
- Read item
- Attach image

[`controller/src/media.rs`](../controller/src/media.rs) defines
`PrivateMediaStore` with byte reads and writes.

The implemented default router currently uses:

- `MemoryCatalogRepository`
- `LocalMediaStore`

This keeps tests and the staged 05-06 deployment local-mode by default.
Production-persistence builds also include:

- `OracleCatalogRepository`, using the native `oracle` crate and Oracle Instant
  Client
- `OciInstancePrincipalMediaStore`, using native OCI Object Storage requests
  signed with runtime instance-principal credentials

The runtime selects them with `AUTOGRAPHS_CONTROLLER_DB_PROVIDER=oracle` and
`AUTOGRAPHS_CONTROLLER_MEDIA_STORAGE_PROVIDER=oci-instance-principal`. The
05-07 live proof validates the full path before public cutover.

## Database Migration

[`controller/db/schema.sql`](../controller/db/schema.sql) is the current
controller-owned Oracle schema. It includes the static-runtime fields that were
originally planned as a migration from the retired Next.js app schema:

- Adds private `original_filename`
- Adds publish-job status tracking
- Adds generated derivative accounting
- Restricts publish mode to `incremental` or `full`
- Restricts derivatives to `thumbnail` or `detail`

## Static Contracts

[`controller/src/contracts.rs`](../controller/src/contracts.rs) defines the
public-safe JSON schema:

- Collection items
- Item details
- Facets
- Public image variants
- Publish manifest

Public media paths look like:

```text
/media/{item-slug}/{image-slug}-thumbnail.webp
/media/{item-slug}/{image-slug}-detail.webp
```

[`controller/src/publisher.rs`](../controller/src/publisher.rs) currently
contains both the fixture contract profiler and the local-mode runtime
publisher. The runtime publisher creates a candidate release, generates
sanitized WebP derivatives, validates the complete public surface, and
atomically promotes a valid release to `current`.

## Current Boundary

The local-mode publisher foundation is implemented. It now:

- Builds complete static candidates from the catalog repository boundary
- Generates sanitized thumbnail and detail WebP derivatives
- Validates JSON, manifest entries, derivatives, byte accounting, and privacy
- Promotes valid candidates atomically through the `current` pointer
- Seeds incremental candidates from `current` and removes stale public output
- Exposes authenticated full, incremental, and status publish endpoints
- Generates minimal static category and tag filtering

The final live static proof and runtime cutover documentation remain separate
05-07 work.

## Static Admin Shell

[`controller/static-admin/`](../controller/static-admin/) contains the minimal
Phase 5 browser shell. It logs in through the HTTP-only cookie flow, creates or
updates metadata, uploads one original with alt text, changes publication
status, triggers incremental or full publishing, and displays redacted publish
status. Phase 6 still owns polished daily-use workflow and media ergonomics.
